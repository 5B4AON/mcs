10.1  What Is Firebase RTDB?

Firebase Realtime Database (RTDB) lets two or more instances of Morse Code Studio exchange decoded Morse characters in real time over the internet. When one station decodes or types a letter, it is instantly relayed to every other station that shares the same channel name and channel secret.

Typical uses:

  • Remote practice — two operators practise CW together without radios or antennas.
  • CW relay — forward decoded off-air CW to a remote operator who can read it in real time.
  • Demonstration — show live Morse decoding on a remote screen or projector at a club meeting.

Firebase RTDB requires an internet connection. Only the last letter per channel is stored — there is no message history or offline caching.

10.2  How Channels Work

A channel is identified by two values:

FieldPurpose
Channel Name A human-readable name (e.g. cw-practice). This is the public identifier — share it freely.
Channel Secret A shared password that restricts who can read or write the channel. Only clients that know both the name and secret can participate.

Both values are used as path segments in the database (/morse-code-studio/channels/{name}/{secret}), so they are effectively a two-part access key. Choose a secret that is hard to guess.

10.3  Input (Receive)

The Firebase RTDB Input section appears in the Inputs tab of Settings. When enabled, the app subscribes to the configured channel and displays incoming characters in the decoder panel and fullscreen conversation view.

Settings

SettingDescription
Enable toggle Start / stop listening to the channel.
Used for Assigns incoming characters to the RX or TX decoder pool. This controls the colour tag in the fullscreen conversation view (same as other input source routing).
Channel Name The channel to subscribe to.
Channel Secret The shared secret for the channel.
Override remote WPM When enabled, incoming characters are played back at your local Encoder WPM instead of the sender’s original speed. Default: off (remote WPM is respected for realistic timing).
Override Name If set, replaces the incoming sender name with this value. Leave empty to preserve the remote sender’s name.
Override Colour If set, overrides the incoming colour. Leave cleared to use the sender’s colour (or the default RX/TX colour).

Each character received from RTDB carries the WPM speed that was used to generate it at the sending station. By default, the receiving app plays back each character at the sender’s original speed — dits, dahs, inter-character gaps, and word spaces are all timed to match the remote operator’s rhythm. This provides a realistic listening experience. If you prefer a constant local speed, enable Override remote WPM to use your Encoder WPM instead.

Received characters are routed to every output whose forward / active-on mode includes RX. By default the sidetone and vibration outputs are set to "Both", so you will hear the incoming morse and feel a haptic pulse. If you also set the optocoupler, serial port or WinKeyer output to "RX" or "Both", the received characters will key those outputs as well — useful for relaying an RTDB-received signal to a transmitter. Characters received from RTDB are never echoed back to the RTDB output, preventing network loops.

10.4  Output (Transmit)

The Firebase RTDB Output section appears in the Outputs tab of Settings. When enabled, each decoded or encoded character is written to the configured channel for remote listeners to receive.

Settings

SettingDescription
Enable toggle Start / stop publishing to the channel.
Channel Name The channel to publish to.
Channel Secret The shared secret for the channel.
User Name Your callsign or name (mandatory). This is shown as a prefix (e.g. [CALLSIGN]) on each line in the remote station's fullscreen conversation view, so they know who is sending. When text is typed via the keyboard encoder, this name is always used.
Override input name When checked, the User Name above is always sent, overriding any input-specific name (e.g. from Mouse Keyer or CW Detector). When unchecked (default), input-specific names are sent if defined; otherwise the User Name is used as fallback.
Colour Optional CSS colour sent alongside each character. Remote listeners display this colour in their fullscreen views. When text is typed via the keyboard encoder, this colour is always used.
Override input colour When checked, the Colour above is always sent, overriding any input-specific colour. When unchecked (default), input-specific colours are sent if defined; otherwise this colour is used as fallback.
Forward Which decoded characters to publish:
  • RX — only received (off-air) characters
  • TX — only transmitted (encoder/keyer) characters
  • Both — all decoded characters

Every character published to RTDB includes the WPM speed used to generate it. For keyboard encoder characters this is the Encoder WPM; for paddle/keyer characters it is the Keyer WPM; for characters decoded from CW audio (RX or TX pools) it is the auto-calibrated estimated WPM. The remote station uses this value to play back each character at its original speed.

10.5  User Name Prefix in Conversation

When Firebase RTDB is active, each line in the fullscreen conversation view is prefixed with the sender's User Name in square brackets, for example: 

[5B4AON] CQ CQ CQ DE 5B4AON K
[G3XYZ] 5B4AON DE G3XYZ GM OM

The prefix uses a smaller, dimmed style so it does not interfere with reading the Morse text. Lines from your own station show your own User Name; lines from remote stations show theirs.

10.6  Bidirectional Setup

For a full two-way conversation, both stations need Firebase RTDB enabled on both Input and Output, using the same channel name and secret. A typical setup:

Station AStation B
Input Channel cw-practice cw-practice
Input Secret my-secret-123 my-secret-123
Input Used for RX RX
Output Channel cw-practice cw-practice
Output Secret my-secret-123 my-secret-123
Output Forward TX TX
Output User Name 5B4AON G3XYZ

With this configuration, each station sees the other's typed text appear in the decoder, colour-coded as RX, with the remote callsign prefix.

10.7  Requirements

  • A Firebase project with Realtime Database enabled. See the Firebase documentation for setup instructions.
  • The Firebase configuration values must be entered in src/app/firebase.config.ts before building the app.
  • An internet connection on both stations. Firebase RTDB does not work offline.
  • Security Rules should be configured in the Firebase Console to validate data shape and prevent abuse. Example rules are documented in firebase.config.ts.

10.8  Auto-Reconnect

If the Firebase RTDB connection drops (network interruption, server error), the app automatically attempts to reconnect using exponential backoff — retrying after 2 s, 4 s, and 8 s. Input (listener) and output (publisher) channels are retried independently.

If all three attempts fail, a persistent amber warning banner appears at the top of the main screen. The banner can be dismissed manually. When the browser comes back online (e.g. after toggling airplane mode or reconnecting to Wi-Fi), the app immediately restarts any input or output channel that should be active, resetting the retry counter.

If you manually disable and re-enable the input or output toggle in Settings, the retry counter is also reset.

10.9  Time Synchronisation & Stale-Character Guard

Firebase RTDB stamps every outgoing character with an authoritative UTC server timestamp provided by Google's infrastructure. When the app connects to Firebase, it also obtains a server-time offset — the difference between your device clock and the Firebase server clock. This works like an NTP (Network Time Protocol) query: the SDK automatically computes the offset so the app always knows the accurate universal time, regardless of your local time zone or clock drift.

If the server-time offset is not available within about one second (very slow network), the app falls back to your device's own clock. Because Date.now() returns UTC milliseconds (timezone-independent), the fallback is still reliable in practice.

Stale-character guard

Firebase RTDB stores the last character written to each channel. When you first subscribe to a channel (or reconnect after a network interruption), the listener immediately receives that stored value — even if it was written hours or days ago.

To prevent old characters from appearing in the decoder, the app performs a one-time freshness check on the very first character received after each connection:

  • The character's server timestamp is compared with the current NTP-synchronised UTC time.
  • If the character is older than one second, it is silently discarded.
  • If the character is fresh (within one second), it is displayed normally.

After the first character passes (or is discarded), all subsequent characters are accepted immediately as they arrive — no further timestamp checks are performed. This ensures zero added latency during normal real-time operation.

This behaviour is fully automatic and transparent — there are no settings to configure.

10.10  RTDB Status Indicator

A small wifi-style icon labelled RTDB appears in the main toolbar and in both fullscreen modal toolbars. It provides at-a-glance feedback about the Firebase connection:

  • Grey — RTDB input and output are both inactive (disabled or not configured).
  • Green — at least one RTDB direction (input listening or output sending) is currently active.

The indicator is purely informational and requires no interaction.
