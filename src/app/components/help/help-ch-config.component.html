8.1  Sound Card Channel Routing

Each stereo sound card has two channels: Left and Right. Morse Code Studio routes each output (sidetone, opto, pilot) to a specific channel on a specific device. This lets you use a single sound card for multiple purposes.

The rule: Two outputs must not share the same channel on the same device. If there is a conflict, the application shows a warning.

Single sound card — choose 2 of 3: A stereo card has only two channels but there are three possible outputs (sidetone, opto, pilot). You can enable any two. To use all three, add a second USB sound card.

8.2  Device Profiles

When you save settings, they are stored against the current set of audio devices connected to your computer. This is called a device profile.

If you later connect a different set of devices (e.g. plugging in a USB sound card at home vs. a portable station), the application detects the change and loads the matching profile automatically. Each hardware configuration is remembered separately.

If no profile matches the current devices, the application creates a new profile with default settings and shows a yellow banner asking you to review and save.

Example — Home vs. portable

At home: desktop PC with a built-in sound card plus a USB card for keying. Portable: laptop with only its built-in audio. Each setup has different device IDs, so Morse Code Studio saves and loads the correct settings automatically.

8.3  Fullscreen Mode

The main toolbar has a ⛶ button that opens a menu with Fullscreen Decoder View and Fullscreen Encoder View options. Choosing either opens a fullscreen overlay optimised for operating.

Independent Display Buffers

The application maintains three independent FIFO text buffers:

  • Main Output — the combined decoder/encoder output panel on the main screen.
  • Fullscreen Decoder — the conversation log in the fullscreen decoder view.
  • Fullscreen Encoder — the conversation log in the fullscreen encoder view.

Each buffer receives text from the same data sources (decoded characters, sent encoder characters, and Firebase RTDB input) but can be cleared independently. Pressing the Clear button (🗑) opens a context menu with This text area (clears the current buffer only) and All text areas (clears all three buffers at once). On the main screen, the encoder’s clear menu adds an Input box option to empty the text-entry field without affecting any log.

Buffers are capped at approximately 5 000 characters (~2× a full screen of text). When the limit is reached, the oldest characters are silently discarded (FIFO). The fullscreen buffers persist across close/reopen — you will not lose your conversation history by switching between views.

Decoder Fullscreen

Large, scrollable decoded text with RX/TX colour coding. The toolbar shows read-only WPM indicator pills (RX, TX, Keyer) and a CW level meter (when CW input is enabled). The kebab menu (⋮) contains interactive WPM adjustment with calibration reset buttons, display settings, and links to the Symbols Reference and Help. When Firebase RTDB input or output is active, a small RTDB status indicator (wifi icon) turns green in the toolbar.

Encoder Fullscreen

Conversation history plus your encoder buffer inline. Every keystroke is captured directly — no need to click a text box. The toolbar shows encoder WPM; the kebab menu (⋮) provides WPM adjustment, sending mode selection (Send on Enter / Live), display settings, and links to the Symbols Reference and Help. On touch-capable devices, a floating keyboard button appears at the bottom to toggle the on-screen virtual keyboard. The RTDB status indicator is also visible here.

Responsive Toolbars

On narrow screens, the main controls bar and each fullscreen modal toolbar automatically collapse overflow controls into a kebab menu (⋮). Essential buttons — Start Audio on the main screen, and Close / Clear in fullscreen modals — remain visible at all times. Tap the ⋮ button to access the remaining controls in a dropdown. A Symbols Ref and Help entry at the bottom of each kebab menu provide quick access to the symbols reference panel and the built-in help system. The toolbar restores to its full inline layout when the window is wide enough.

Display Customisation

The following display settings are available in both fullscreen views and are persisted independently of device profiles:

  • Font size — Adjustable from 24 to 184 px.
  • Line spacing — Vertical spacing between conversation lines.
  • Colours — Custom colours for RX text, TX text, buffered text, currently-sending character, and background.

8.4  Text Blurring

Found in Settings → Other. When enabled, decoded and/or encoded text is blurred in both the main screen and fullscreen views, turning Morse Code Studio into a copy-receiving trainer. You hear the morse and write what you think was sent, then reveal the answer to check.

The Applies to selector controls which text streams are blurred:

  • RX Only — only received (decoded) text is blurred.
  • TX Only — only transmitted (encoded) text is blurred.
  • Both — all text is blurred.

To reveal the hidden text, press and hold the eye button (👁️) that appears when blurring is active. The text fades in with a 250 ms animation while you hold the button and blurs again when you release it. In the fullscreen conversation views, user-name tags from RTDB relay messages remain visible even when the text is blurred, so you always know who is transmitting.

8.5  Show Prosigns

Found in Settings → Other. When enabled, punctuation characters that share morse patterns with prosigns are displayed as their prosign names in the fullscreen conversation logs. For example, + appears as <AR>, and = appears as <BT>.

This is a display-only transformation — the application continues to work with punctuation characters internally. The conversion applies to the fullscreen decoder and encoder conversation windows only; main screen log panels show the raw punctuation. Prosigns that do not share a pattern with punctuation (e.g. <SK>, <HH>) are always shown in their prosign form regardless of this setting.

8.6  Prosign Actions

Found in Settings → Other. When the master toggle is enabled, decoded prosigns automatically trigger formatting actions in the fullscreen conversation logs.

Four prosigns are supported, each individually enabled and mapped to one of five actions:

Prosign Default Action Description
<AR> New Paragraph End of message — inserts two blank lines
<BK> New Line Break — inserts a line break
<BT> New Line New paragraph — inserts a line break
<HH> Clear Last Word Error — removes text back to the last space or newline boundary

Available actions:

  • New Line — inserts a single line break.
  • New Paragraph — inserts two line breaks (blank line separator).
  • Clear Last Word — removes text back to the last space or newline boundary (or the start of the buffer if there is none).
  • Clear Line — removes all text on the current line back to the previous newline (or the start of the buffer).
  • Clear Screen — clears the current fullscreen buffer entirely. Other buffers (main panel, the other fullscreen view) are not affected.

For the New Line and New Paragraph actions the prosign notation (e.g. <BT>, <AR>) is shown at the end of the current line before the line break is inserted, so you can see which prosign triggered the formatting. Destructive actions (Clear Last Word, Clear Line, Clear Screen) do not display the prosign.

Prosign actions only affect the fullscreen decoder and encoder conversation logs. The main screen panels always show the raw decoded text. The word-gap space that the decoder normally emits after a character is automatically suppressed when a prosign action fires, so new lines start without a leading space.

8.7  Emoji Replacements

When enabled, emoji replacements substitute common morse abbreviations, prosigns, Q-codes, and 92 codes with emoji graphics in the fullscreen decoder and encoder conversation logs. This adds visual context at a glance — a greeting, a farewell, a signal report — without affecting the underlying text or the morse encoding itself.

Each mapping has a match pattern (case-insensitive), a replacement, and an optional meaning description. Matching is whole-word only: the pattern must appear at a word boundary on both sides (start of text, space, or newline). Prosign patterns such as <AR> also match their punctuation equivalents (e.g. +).

Match patterns can be single characters (e.g. +), prosigns (e.g. <AR>), or multi-character sequences (e.g. TNX, 73, QSL). The replacement text is not limited to a single emoji — it can be any mix of emojis and/or plain characters. For example, the default mapping for <KN> uses 🎤🔒 (two emojis), and you could map CQ to 📡 CQ (emoji followed by text).

The feature ships with a curated set of defaults (disabled by default). You can enable it, add your own mappings, reorder, or delete entries in Settings → Other → Emoji Replacements.

Example — Sample QSO with emojis & prosign actions

Copy and paste the text below into the encoder to see how prosign actions and emoji replacements work together in the fullscreen view:

CQ CQ CQ DE 5B4AON 5B4AON K
GA OM TNX FER CALL UR RST 599 599 QTH CYPRUS
WX IS FB HERE HI
QSL VIA LOTW GL 73 88 <SK>
