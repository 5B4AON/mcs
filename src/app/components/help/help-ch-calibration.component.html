5.1  What Are RX and TX?

Every input in Morse Code Studio feeds its timing events into the decoder. The decoder needs to know whether each input represents received Morse (from another operator) or transmitted Morse (your own sending). This distinction is called the decoder source and it determines two things:

  1. Calibration pool — Which set of adaptive timing averages the input feeds into (RX pool or TX pool).
  2. Colour tag — Whether decoded text appears as RX (green by default) or TX (yellow by default) in the fullscreen conversation view.

5.2  Per-Input Source Assignment

Each input type has a "Used for:" selector in its Settings section. You can set each one independently to either RX or TX. The Keyboard Keyer, MIDI Input, and Serial Input allow separate assignment for straight key and paddle:

A colour-coded badge in each Settings section header shows the current assignment at a glance: blue for RX, amber for TX.

5.3  Default Assignments and Reasoning

The defaults are chosen for the most common use case: the CW tone detector, MIDI input, serial input and pilot-tone mic input default to RX because hardware inputs typically carry Morse from another operator (a radio receiver feeding audio, or a remote station's key connected for demonstration or relay). Keyboard, mouse and touch keyers default to TX because they represent your own sending.

However, you may want to change these assignments in certain scenarios:

  • Practice with a physical key: If you connect a straight key via MIDI, serial, or the pilot tone and want to practise your own sending, set the input to TX so your keying shows as TX text and calibrates the TX pool.
  • Remote keyer: If someone is keying remotely through an audio feed, their input is RX even though it comes through what might normally be a "local" source.

5.4  Why Separate Pools Matter

Without separate calibration pools, your 25 WPM keyboard keying would be averaged together with a 15 WPM signal coming from the radio. The threshold would drift to somewhere around 20 WPM, causing the decoder to misclassify dahs as dits (or vice versa) for both sources.

With dual pools, each source maintains its own accurate speed estimate regardless of what the other is doing. The decoder always uses the correct pool's threshold when classifying elements from a given input.

Example — QSO at different speeds

You are sending at 22 WPM using the keyboard keyer (TX pool). The station you are working sends at 14 WPM via your radio's CW tone input (RX pool). Each pool auto-calibrates independently: TX settles at ~22 WPM, RX at ~14 WPM. Both decode correctly without interfering with each other.
