6.1  Decode from Audio Channel (CW Tone Detector)

This input decodes CW audio tones directly from a radio receiver, virtual audio cable, or any other source carrying CW tones. No external circuit is needed — simply route audio into a sound card input. This is the primary decode path for most setups and works reliably across all supported platforms.

How It Works

A narrow-band Goertzel filter is tuned to the configured CW frequency (default 600 Hz). When the tone amplitude exceeds the threshold, the key is considered down.

Settings

  • Audio Input Device — The sound card receiving radio audio.
  • Input Channel — Left or Right.
  • CW Tone Frequency — Match your radio's sidetone / BFO offset. Typically 500–700 Hz.
  • CW Tone Bandwidth — Width of the detection passband in Hz (100–500). For example, at 600 Hz centre frequency and 100 Hz bandwidth the passband is 550–650 Hz. Narrower bandwidth is more selective against nearby interference but requires a more precise frequency setting. Default: 100 Hz.
  • Auto Threshold — When enabled, the detector automatically tracks the noise floor and signal level, adjusting the threshold dynamically. Recommended for most use.
  • Manual Threshold — Available when Auto Threshold is off. Set by hand using the slider for fixed-level signals.
  • Debounce — Filters brief noise spikes. 5–10 ms is typical for clean signals.
  • Used for — Decoder source (RX or TX). See Chapter 5: Decoder Source Routing.
Example — Decoding with a virtual audio cable

On Windows, install "VB-Cable" or "Virtual Audio Cable". Route your SDR receiver's audio output to the virtual cable. In Morse Code Studio, select the virtual cable as the Audio Input Device. Set the CW frequency to match your SDR's BFO offset. Enable Auto Threshold. The decoder now shows decoded CW text live.

6.2  MIDI Input

Connect a MIDI pedal, foot switch, keyboard, or any MIDI controller to your computer and use it as a morse key or paddles. MIDI input has a key advantage over keyboard and mouse input: it works even when the app is in the background, another window is focused, or the screen is locked / showing a screensaver.

How It Works

The Web MIDI API delivers note-on and note-off messages from MIDI hardware at the browser level, independent of DOM focus. Morse Code Studio maps specific MIDI notes to keyer actions — straight key, dit paddle, or dah paddle — using the same keyer engine as the keyboard and mouse keyers.

Settings

  • MIDI Device — Select a specific MIDI input, or leave on “Any” to accept messages from the first available device.
  • MIDI Channel — Filter by MIDI channel (1–16), or “Omni” to accept all channels.
  • Straight Key — Click the capture button and press a key/pad on your MIDI controller. That note will act as a straight key (press = key down, release = key up).
  • Dit Paddle — Capture a MIDI note for the dit paddle. Used with iambic keyer modes.
  • Dah Paddle — Capture a MIDI note for the dah paddle.
  • Reverse Paddles — Swap dit and dah paddle assignments.
  • Paddle Mode — Iambic A, Iambic B, Ultimatic, or Single Lever (shared with keyboard and mouse keyers).
  • Used for — Decoder source (RX or TX). See Chapter 5: Decoder Source Routing.
Assigning MIDI notes

1. Enable MIDI Input and expand the settings card.
2. Click the “Straight Key” capture button — it shows “Press a MIDI key…”.
3. Press the desired key or pedal on your MIDI controller.
4. The button updates to show the captured note name (e.g. “C4 (60)”).
5. Repeat for Dit and Dah paddles if using paddle mode.
6. Click the × button next to any assignment to clear it.

Background operation

Unlike keyboard or mouse input, MIDI messages are delivered by the browser even when the tab is not focused. Combined with the audio keep-alive that Morse Code Studio maintains, your MIDI key/pedal continues to work during screensaver, lock screen, or while another application has focus.

MIDI input also bypasses the keyboard keyer’s enabled/disabled state. When a text field has focus, the keyboard keyer is automatically disabled to avoid stray characters, but MIDI keying continues uninterrupted. This means you can type in the encoder while still keying via your MIDI pedal or controller.

Full-duplex echo prevention & break-in (QSK)

When MIDI input and output share the same bus or device, the app automatically prevents feedback loops at two independent levels:

1. Hardware-level: isSending muting

  • Echo suppression — while a MIDI output note is physically held, all incoming note-on messages are treated as loopback echoes and dropped.
  • Break-in — when a valid (non-echo) MIDI input note arrives, the entire MIDI output queue is flushed, held notes are released, and output is muted. New characters from the encoder are dropped until the remote party goes silent for a full word gap, at which point output resumes automatically.

2. Software-level: Decoder output routing

  • Receive pipeline classification — MIDI input events are tagged with fromMidi and classified as “receive” pipelines in the decoder. Receive pipelines activate only sidetone and vibration for monitoring. They do not activate serial output or MIDI output through the real-time keyDown/keyUp path. This prevents a software feedback loop where a MIDI key press would trigger the serial or MIDI output, which could then echo back.
  • Reverse-path tagging — characters decoded from MIDI input carry a fromMidi tag and are never forwarded back to MIDI output or serial output through the character forwarding path.

Together, these two layers ensure that MIDI input can safely coexist with any combination of outputs — even when both use the same bus or source routing (RX/TX). The same pattern applies to Serial Input (§6.3).

Browser compatibility

Web MIDI API is supported in Chrome, Edge, and Opera. Firefox and Safari do not support it. MIDI permission is requested transparently when you click Start Audio — no separate permission step is needed.

Arduino Pro Micro as a MIDI key/paddle

The project includes ready-made Arduino sketches (in the arduino/ folder of the source repository) that turn a Pro Micro into a USB MIDI device with straight key and paddle inputs on pins 2, 3 and 4. Two board variants are provided: ATmega32U4 (classic Pro Micro, using the MIDIUSB library) and nRF52840 (Supermini / nice!nano, using Adafruit TinyUSB). Flash the sketch, connect your key or paddles, and the Arduino appears as a standard MIDI device — no drivers needed. Wiring diagrams and setup instructions are included.

6.3  Serial Input (DSR/CTS/DCD/RI)

Connect a straight key or paddles through a USB-to-serial adapter (FTDI, CH340, CP2102, etc.) and use the adapter’s input status signals — DSR, CTS, DCD, and RI — as keying inputs. This method provides low-latency, hardware-level key detection without any audio processing.

How It Works

USB-to-serial adapters expose several control/status signal pins. Two of these pins — DTR and RTS — are output pins controlled by the computer (used by Serial Output for transmitter keying). The remaining four — DSR, CTS, DCD, and RI — are input pins that the computer reads. By connecting a Morse key between one of these input pins and ground (GND), closing the key pulls the pin low, which the app detects.

The app polls the input signals via getSignals() at a configurable interval (default 10 ms). Per-signal debounce (default 5 ms) filters contact bounce from mechanical keys. The combination of poll interval and debounce determines the maximum usable keying speed.

Port Sharing

If Serial Output is already using the same USB-serial adapter, Serial Input automatically “piggybacks” on the open port — no separate connection is needed. This allows you to use DTR or RTS for transmitter keying and DSR/CTS/DCD/RI for key input on the same adapter simultaneously.

Settings

  • Serial Port — Select the USB-serial adapter. Click “Add Serial Port…” to grant access to a new port. Previously granted ports are remembered across page refreshes.
  • Poll Interval — How often the signal pins are read, in milliseconds (5–50 ms). Lower = faster response but more CPU usage. Default: 10 ms.
  • Debounce — Minimum time a signal must remain stable before a state change is accepted (2–10 ms). Filters contact bounce. Default: 5 ms.

Straight Key

  • Signal Pin — Which input pin to use: DSR, CTS, DCD, or RI.
  • Used for — Decoder source (RX or TX). See Chapter 5: Decoder Source Routing.
  • Invert — Swap active-high/active-low logic. Most adapters assert a signal when the pin is grounded, so the default (not inverted) treats grounded = key down. If your wiring uses the opposite convention, enable invert.

Paddle

  • Dit Paddle Pin — Input pin for the dit paddle (default: CTS).
  • Dah Paddle Pin — Input pin for the dah paddle (default: DCD).
  • Used for — Decoder source (RX or TX).
  • Invert — Invert both paddle signals.
  • Reverse Paddles — Swap dit and dah.
  • Paddle Mode — Iambic A, Iambic B, Ultimatic, or Single Lever (shared with other keyer inputs).
Wiring a straight key

Connect one terminal of your key to the adapter’s DSR pin, and the other terminal to GND. In settings, select DSR as the straight key pin. Close the key — the DSR LED should light up and the decoder should register a key-down.

Wiring paddles

Connect the dit paddle between CTS and GND, and the dah paddle between DCD and GND. The common wire goes to GND. In settings, set Dit Pin = CTS and Dah Pin = DCD. The iambic keyer generates correctly timed elements at the configured WPM speed.

Simultaneous input and output

A single USB-serial adapter can serve as both input and output at the same time. Set Serial Output to key via DTR (or RTS), and Serial Input to read paddles on CTS + DCD. Select the same port index in both cards. The app shares the port automatically.

Echo suppression & feedback prevention

When Serial Input and Serial Output share the same adapter (or are cross-wired), the app automatically prevents feedback loops at two independent levels:

1. Hardware-level: isSending muting

  • Output mute — while the Serial Output is actively keying (DTR/RTS asserted), all incoming signal changes on the input pins are suppressed. This prevents cross-talk between output and input pins — especially on adapters where toggling DTR/RTS can momentarily glitch DSR/CTS readings.
  • 30 ms holdoff — after the output releases the keying pin, input remains suppressed for an additional 30 ms to allow the bus to settle. This eliminates trailing edge echoes caused by cable capacitance and adapter settling time.

2. Software-level: Decoder output routing

  • Receive pipeline classification — Serial input events are tagged with fromSerial and classified as “receive” pipelines in the decoder. Receive pipelines activate only sidetone and vibration for monitoring. They do not activate serial output or MIDI output through the real-time keyDown/keyUp path. Without this, pressing a serial key would trigger the serial output (asserting DTR/RTS), creating an immediate software feedback loop — even when the hardware isSending muting would eventually suppress the echo.
  • Reverse-path tagging — characters decoded from serial input carry a fromSerial tag and are never forwarded back to MIDI output through the character forwarding path. This prevents decoded serial input from echoing onto the MIDI bus.
  • Independent keyer — Serial Input has its own self-contained iambic keyer that calls the decoder directly, completely independent of the keyboard/mouse/touch keyer. This prevents shared-state conflicts between input sources.

Together, these two layers ensure that Serial Input can safely coexist with Serial Output and MIDI Output — even when using the same adapter and the same source routing (RX/TX). The same pattern applies to MIDI Input (§6.2).

Maximum keying speed

With the default settings (10 ms poll, 5 ms debounce), the maximum practical speed is about 60 WPM. At 5 ms poll and 2 ms debounce, you can reach 120 WPM. The card shows the estimated maximum WPM based on your current settings.

FTDI adapters vs other chips

FTDI-based adapters (FT232R, FT232H, FT2232) typically expose all four input pins as solder pads or header pins. CH340 and CP2102 adapters may only break out DSR and CTS. Check your adapter’s documentation for available pins.

Browser compatibility

Web Serial API is supported in Chrome and Edge on desktop (Windows, macOS, Linux, ChromeOS). It is not available on Android, iOS, Firefox, or Safari. Serial Input requires the same browser support as Serial Output.

6.4  Straight Key via Mic (Pilot Tone) Experimental

This input method connects a physical Morse key (straight key or bug) through the microphone input of your sound card using an ultrasonic pilot tone. This feature is experimental — AC coupling behaviour varies across sound cards and USB audio adapters, and results may be unreliable on some hardware.

How It Works

  1. The application sends a constant high-frequency tone (default 18 kHz — inaudible to most people) out of one audio channel to a simple external circuit.
  2. The circuit routes the tone through your Morse key and into the mic input.
  3. When the key is closed, the tone reaches the microphone. When the key is open — silence. A Goertzel filter detects the tone with high precision.

Settings

  • Mic Device — The microphone or sound card input that receives the pilot tone.
  • Pilot Output Device — The sound card that generates the pilot tone. Can be the same card or a different one.
  • Pilot Output Channel — Left or Right. Must not conflict with sidetone or opto on the same device.
  • Pilot Frequency — Default 18,000 Hz (above human hearing). Lower this if your sound card does not pass 18 kHz cleanly.
  • Pilot Amplitude — Volume of the pilot tone. Usually 1.0 is best for reliable detection.
  • Threshold — Detection threshold. Use the Calibrate buttons (described below) to set automatically.
  • Debounce — Minimum time (ms) between state changes. Filters contact bounce from mechanical keys. 5–15 ms is typical.
  • Invert key sense — Swap key-open/key-closed. Useful if your wiring is inverted.
  • Used for — Decoder source (RX or TX). See Chapter 5: Decoder Source Routing.
Calibrating the threshold

1. Click Start Audio. 2. Ensure your key is open (not pressed) and click 1. Key Open — wait for the measurement. 3. Close the key (press and hold) and click 2. Key Closed. 4. Click Apply threshold. The threshold is now set midway between the open and closed levels.
