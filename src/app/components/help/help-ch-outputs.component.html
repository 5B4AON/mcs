7.1  Key via Audio Channel (Optocoupler) Experimental

This output keys your transmitter through an optocoupler driven by a sound card audio channel. The optocoupler provides electrical isolation between your computer and radio — important for safety and to prevent ground loops. This feature is experimental — AC coupling on many sound cards distorts the DC component of the keying signal, which can produce unreliable timing. USB-serial keying (§7.2) is generally more reliable.

Drive Modes

ModeSignalBest for
DC Offset Steady DC voltage when keyed, zero when not keyed. USB sound cards that pass DC. Simplest wiring — try this first.
AC Tone Square wave at the configured frequency when keyed. Sound cards that block DC (capacitor-coupled outputs).

Settings

  • Audio Device — The sound card driving the optocoupler.
  • Channel — Left or Right. Must not conflict with sidetone or pilot tone on the same device.
  • Mode — DC Offset or AC Tone.
  • Frequency — (AC mode only) Square wave frequency. 400–1000 Hz usually works well.
  • Amplitude — Set to 1.0 for maximum drive voltage.
  • Forward — Controls which signals activate the optocoupler:
    • TX only (default) — Fires only for transmitted morse (encoder, keyer inputs tagged as TX).
    • RX only — Fires only for received morse (decoded audio inputs, Firebase RTDB).
    • Both — Fires for all morse activity.
  • Test Output — Keys the output for 1 second to verify wiring.
Example — Simple DC wiring

Get a PC817 or 4N25 optocoupler. Connect the sound card's audio output to the LED anode (pin 1) and GND to the cathode (pin 2). Connect the phototransistor pins (4 and 5) to your radio's key jack. Set mode to DC Offset, amplitude 1.0. Click "Test Output" — your radio should key.

7.2  Key via Serial Port (DTR/RTS)

This output keys your transmitter using the DTR or RTS control line of a USB-serial adapter. These lines switch in under a millisecond, making them ideal for CW keying at any speed.

Requirements

  • A Chromium-based browser (Chrome or Edge) — Firefox and Safari do not support the Web Serial API.
  • A desktop operating system (Windows, macOS, Linux or ChromeOS). Android is not supported — Chrome on Android does not provide access to USB-serial adapters, even when connected via USB OTG. The Web Serial API flag exists in Android Chrome but USB-to-serial devices are not enumerated.
  • A USB-serial adapter such as FTDI FT232, CH340, CP2102 or similar.
  • An optocoupler or transistor between the adapter and your radio, unless your radio accepts ±5–12 V keying directly.

Multiple Mappings

Serial Output supports multiple independent mappings in a table-based interface (the same pattern as MIDI Output §7.4). Each mapping defines its own serial port, output pin (DTR or RTS), invert setting, and forward selector. You can:

  • Add a new mapping with the + Add Mapping button.
  • Edit any mapping via the pencil button to open the edit modal.
  • Remove a mapping with the delete button in the edit modal.
  • Enable / disable individual mappings with the checkbox.

This allows you to key multiple transmitters or drive multiple output pins simultaneously — for example, one mapping keying a transmitter via DTR on one adapter, while another mapping drives a relay via RTS on a second adapter. You can also target both DTR and RTS on the same adapter with two separate mappings.

Per-Mapping Forward Selector

Each mapping has its own Forward setting that controls which signal sources drive it:

  • TX only (default) — Keys only for transmitted morse (encoder, keyer inputs tagged as TX).
  • RX only — Keys only for received morse (decoded audio inputs, Firebase RTDB).
  • Both — Keys for all morse activity.

This per-mapping control lets you route RX and TX to different serial adapters or pins independently.

Settings (per mapping)

  • Serial Port — Click “Add Serial Port…” to grant access to a new adapter, or select a previously granted port. You must grant access each time the page loads.
  • Output Pin — DTR (Data Terminal Ready) or RTS (Request To Send). Either works; DTR is more common for CW keying.
  • Invert Output — By default, the physical pin is held LOW when idle and driven HIGH when keyed. The HIGH pulse activates external hardware (e.g. an optocoupler). When checked, the polarity is reversed (idle = HIGH, keyed = LOW).
  • Forward — Which signal sources drive this mapping: TX only, RX only, or Both.
  • Test Output — Keys the output for 1 second to verify wiring. Available when the mapping’s port is connected.

FTDI / CH340 Pin Behaviour

USB-serial adapter chips (FTDI, CH340, CP2102, etc.) use active-low DTR# and RTS# outputs internally. This means the physical DTR and RTS pins default to HIGH when the adapter is first plugged in or when the operating system opens the serial port. High is also the state that engages an optocoupler wired between DTR and ground, so a newly connected adapter will key your transmitter continuously until software takes control.

Morse Code Studio reverses this initial state: as soon as Serial Output is enabled, the app drives the pins LOW (the idle state by default), silencing the output. However, there is an unavoidable moment between opening the port and the app’s first setSignals() call where the pins remain HIGH.

To avoid inadvertently keying your transmitter during this brief window, follow this connection procedure:

  1. Connect the USB-serial adapter to your computer and select it in the Serial Output settings — do not attach any external hardware to the adapter yet.
  2. Enable Serial Output — the app opens the port and immediately drives the pins LOW. The brief HIGH glitch during port.open() is harmless because nothing is connected.
  3. Now connect your external hardware (optocoupler, transistor keying circuit, etc.) to the adapter’s DTR/RTS pins. The pins are already LOW and idle, so the transmitter will remain silent until you key.

When disconnecting, reverse the order: detach the external hardware first, then disable Serial Output. This avoids the same HIGH glitch in the opposite direction.

Example — FTDI adapter keying

Plug an FTDI FT232 USB adapter in. Enable Serial Output and click the pencil button on the default mapping. Click “Add Serial Port…” and select the adapter. Set Output Pin to DTR. Wire DTR through a 1 kΩ resistor to the base of an NPN transistor (2N2222). Collector to radio key line, emitter to ground. Click “Save” and then “Test Output”.

Echo Suppression (when sharing port with Serial Input)

When Serial Output and Serial Input (§6.3) share the same adapter, feedback is prevented at two levels. The holdoff is tracked per mapping, so mappings on different ports operate independently.

Hardware-level: isSending muting

The output tracks an isSending state that automatically mutes serial input while keying is active:

  • On key-down, the output sets isSending to true. All incoming pin changes on the Serial Input are suppressed while this flag is set.
  • On key-up, a 30 ms holdoff timer keeps isSending true, allowing the serial bus to fully settle before input is unmuted. If another key-down arrives during the holdoff, the timer is cancelled and isSending stays true continuously.
  • For timed pulses (encoder playback), the same holdoff applies after each pulse completes.

Software-level: Decoder output routing

Even without hardware cross-talk, a software feedback loop would occur if serial input events could activate serial output through the decoder. Serial input events are tagged fromSerial and classified as “receive” pipelines, which activate only sidetone and vibration — never serial or MIDI output. Characters decoded from serial input are also tagged and excluded from the character forwarding path to MIDI output.

This dual-layer approach (same as MIDI Output §7.5) eliminates both hardware cross-talk loops and software routing loops.

Reconnection

Serial Output connections are restored automatically on page load when previously enabled. If a USB adapter is unplugged and replugged, the app detects the new device and attempts to reconnect. When the port reappears at a different index in the browser’s port list, the app uses the stored USB Vendor/Product ID to relocate it.

Browser limitation: Whether the browser remembers a previously granted port after disconnect depends on the device’s USB descriptor. Adapters with a unique USB serial number (e.g. genuine FTDI chips) are reliably matched. Some budget chipsets (e.g. CH340 clones without a programmed serial number) may not be recognised by the browser after re-plug, in which case you need to re-grant the port via the + Add serial port… dropdown option. This is an inherent Web Serial API / Chromium limitation, not an application issue.

7.3  WinKeyer Output (K1EL WinKeyer)

This output forwards decoded text to a K1EL WinKeyer device (WK2, WK3 or WKUSB) connected via a virtual COM port. WinKeyer generates perfectly timed CW keying on its output pin — the host simply sends plain ASCII characters and WinKeyer handles the morse element timing internally at the configured WPM speed.

How It Works

Morse Code Studio decodes incoming key events (from any input source) into text characters tagged as RX (received morse) or TX (transmitted morse). The WinKeyer output watches for these decoded characters and forwards them to the WinKeyer device, which re-keys the morse with clean timing. The Forward setting controls which characters are relayed:

  • RX only — Only characters decoded from incoming audio inputs (mic pilot tone, CW tone detector) or inputs configured as RX.
  • TX only — Only characters from the encoder (typed text) and from keyer inputs (keyboard, mouse, touch) configured as TX.
  • Both — All decoded characters regardless of source.

Connection

WinKeyer uses 1200 baud, 8 data bits, no parity, 2 stop bits (8N2). The WKUSB or a WinKeyer IC with a USB-serial adapter appears as a virtual COM port. Morse Code Studio uses the Web Serial API to communicate with it, so a Chromium-based browser (Chrome or Edge) on a desktop operating system is required. Android is not supported — USB-to-serial devices like WinKeyer are not accessible from Chrome on Android, even via USB OTG.

Settings

  • Add Serial Port — Opens the browser's serial port picker. Select the WinKeyer's COM port.
  • WPM Speed — Sets WinKeyer's internal keying speed (5–99 WPM). Changes are sent to the device in real time.
  • Forward — Which decoded text to relay: RX only, TX only, or Both.
  • Test — Sends the word "TEST" to WinKeyer to verify the connection.
  • Clear Buffer — Aborts any pending text in WinKeyer's internal transmit buffer.
Example — Re-keying received morse

Connect a WKUSB to your computer. In Outputs → WinKeyer Output, click "Add Serial Port" and select the WKUSB COM port. Set Forward to "RX only" and WPM to match the incoming station. Enable WinKeyer output. Received morse from your CW audio input or mic is decoded, forwarded to WinKeyer, and re-keyed with clean timing on WinKeyer's output — useful for relay, practice copying, or driving a second transmitter.

Example — Clean TX keying

Set Forward to "TX only". Type text in the Encoder panel and send it. The typed characters are forwarded to WinKeyer, which generates perfectly timed CW keying — ideal when you want WinKeyer's hardware timing precision for your transmitted morse rather than software-generated audio keying.

Reconnection

WinKeyer auto-connects on page load when previously enabled, and attempts to reconnect when the USB cable is unplugged and replugged. Retries are sent at increasing intervals (0, 500, 1500, 3000 ms) to handle slow USB enumeration. If the port reappears at a different index, the app uses the stored USB Vendor/Product ID to find it.

Browser limitation: Whether the browser remembers a previously granted port after disconnect depends on the device’s USB descriptor. WKUSB uses an FTDI chip with a unique serial number and is reliably matched in most cases. If the browser does not restore the grant (e.g. after clearing site data), re-grant the port using the + Add serial port… dropdown option.

7.4  MIDI Output (Key via MIDI)

The MIDI output sends note-on and note-off messages to a MIDI device for each decoded character's morse elements. Unlike the serial and audio outputs which key in real time with the input, MIDI output is character-based — it receives decoded characters and plays them back as individual dits and dahs using the Encoder WPM timing setting. This is similar to how WinKeyer output (§7.3) works: the application sends characters, and the output produces properly timed morse elements.

Multiple Mappings

MIDI Output supports multiple independent mappings in a table-based interface (the same pattern as MIDI Input §6.2). Each mapping defines its own MIDI device, channel, mode (straight key or paddle), note assignments, and forward selector. You can:

  • Add a new mapping with the + Add button.
  • Edit any mapping via the pencil button to open the edit modal.
  • Duplicate an existing mapping as a starting point.
  • Remove a mapping with the delete button.
  • Enable / disable individual mappings with the checkbox.

This allows you to drive multiple MIDI devices (or multiple channels on the same device) simultaneously — for example, one mapping keying a transmitter via an Arduino optocoupler on channel 1, while another mapping drives an LED indicator on channel 2.

Per-Mapping Forward Selector

Each mapping has its own Forward setting that controls which signal sources drive it:

  • TX only (default) — Fires only for transmitted morse (encoder, keyer inputs tagged as TX).
  • RX only — Fires only for received morse (decoded audio inputs, Firebase RTDB).
  • Both — Fires for all morse activity.

This per-mapping control lets you route RX and TX to different MIDI devices or channels independently.

All Input Sources Supported

MIDI output works with every input type:

  • Encoder (typed text) — Characters are forwarded directly from the encoder send loop for real-time TX playback.
  • Keyer paddles (keyboard, mouse, touch) — Characters are decoded first, then forwarded and played back at encoder WPM timing.
  • Straight key (keyboard, mouse, touch) — The operator's raw keying is decoded into characters, which are then played back as clean dits and dahs at encoder WPM — effectively "cleaning up" the timing.
  • Audio inputs (mic pilot tone, CW tone detector) — Decoded characters are played back at encoder WPM.
  • MIDI input — Decoded characters from incoming MIDI are played back on the MIDI output.
  • Firebase RTDB — Characters received from a remote station are played back at the sender’s original WPM by default, preserving the remote operator’s speed. Enable “Override remote WPM” to use your local Encoder WPM instead.

Intended Use Case

Connect an Arduino Pro Micro (or similar ATmega32U4 board) running the MIDIUSB library. The Arduino appears as a native USB MIDI device — no drivers needed. The Arduino sketch monitors incoming MIDI notes and maps them to digital output pins, which in turn drive keying circuits, paddle jacks, or relay boards. Combined with MIDI Input (§6.2), the same Arduino can provide bidirectional real-time interfacing with the application.

Ready-made sketches for both the classic ATmega32U4 Pro Micro (using MIDIUSB) and the nRF52840 variant (Supermini / nice!nano, using Adafruit TinyUSB) are included in the arduino/ folder of the source repository, along with wiring diagrams and setup instructions. The v1.1.0 sketches support 16 configurable pins (10 inputs, 6 outputs by default) across two MIDI channels.

Output Modes

Each mapping is configured as either Straight Key or Paddle mode:

  • Straight Key — A single MIDI note fires during every element (dit or dah) and releases during the gap between elements. Provides a simple “key down / key up” signal for keying circuits.
  • Paddle — Two MIDI notes: one for dit elements, one for dah elements. The dit note fires during dit elements, the dah note fires during dah elements. This allows the Arduino to distinguish dit from dah for advanced control (e.g. driving separate paddle jacks).

Both modes can coexist — you can have a straight-key mapping and a paddle mapping enabled simultaneously. During a dit element, the straight-key note and the dit paddle note fire together. During a dah element, the straight-key note and the dah paddle note fire together.

Independent Lifecycle

MIDI output starts independently when enabled — it is not tied to the audio start/stop button. This avoids issues with USB MIDI devices (which can become locked when MIDI access is rapidly torn down and re-requested, such as during a browser refresh). The browser remembers the MIDI access grant across sessions, so MIDI output reconnects automatically on page load.

Settings (per mapping)

  • MIDI Device — Select the target MIDI output device. “Any (first available)” uses the first connected output.
  • MIDI Channel — Output channel (1–16).
  • Forward — Which signal sources drive this mapping: TX only, RX only, or Both.
  • Mode — Straight key or Paddle.
  • Note — The MIDI note for straight key or dit paddle. Use the note/octave picker or raw 0–127 entry.
  • Dah Note — (Paddle mode only) The MIDI note for the dah paddle.
  • Override remote WPM with local encoder WPM — Characters received via Firebase RTDB carry the sender’s original WPM speed. By default MIDI output uses that remote speed so the playback matches the sender’s timing. Enable this option to ignore the remote WPM and use your local Encoder WPM instead.
  • Test — Sends a 1-second note-on/off on the mapping’s note to verify the connection.

Browser Requirements

  • Requires a Chromium-based browser (Chrome, Edge, Opera). Firefox and Safari do not support the Web MIDI API.
  • The browser prompts for MIDI access permission the first time. The grant is remembered across sessions.

Break-In (QSK)

When MIDI Input and MIDI Output share the same bus, MIDI output supports automatic full break-in. When a valid (non-echo) note arrives on MIDI input, MIDI output immediately:

  1. Aborts the current element playback and flushes all queued characters.
  2. Releases any held notes so the bus is silent.
  3. Enters a muted state — new characters from the encoder are dropped silently.

The muted state persists until the remote party’s input goes silent for a full word gap (7 dit units at the current encoder WPM). After the word gap expires, MIDI output resumes accepting characters normally.

Decoder Output Routing

In addition to the hardware-level echo suppression above, the decoder prevents software feedback loops by classifying MIDI input events as “receive” pipelines. Receive pipelines activate only sidetone and vibration — they do not activate MIDI output or serial output through the real-time keyDown/keyUp path. Characters decoded from MIDI input are also tagged fromMidi and excluded from the character forwarding path to MIDI output. This dual-layer design (same as Serial Output §7.2) eliminates both hardware bus echoes and software routing loops.

Example — Arduino Pro Micro MIDI keyer

Flash an Arduino Pro Micro with the v1.1.0 MIDIUSB sketch. The sketch drives output pins 11–16 based on incoming MIDI notes. In Outputs → MIDI Output, the default mappings already match the Arduino’s default output notes: straight key on G♯5 (80) and paddle on A♯5/C6 (82/84), all on channel 1. Select the Arduino MIDI device in each mapping and enable MIDI Output. When you send text via the Encoder or key with any input, the decoded characters are played back as properly timed morse elements on the Arduino’s output pins.

Note: The output notes (80/82/84) deliberately differ from the input notes (60/62/64) used for MIDI Input (§6.2). This prevents a feedback loop when both MIDI Input and MIDI Output are enabled on the same device simultaneously.

Example — Bidirectional MIDI interface

Enable both MIDI Input (§6.2) and MIDI Output on the same Arduino. The Arduino reads physical paddle switches and sends MIDI notes to the app (input on notes 60/62/64), while the app decodes those key events into characters and plays them back on different MIDI output notes (80/82/84) to drive an LED, buzzer, or relay. Using separate note numbers for input and output prevents feedback loops. This creates a complete hardware interface with no serial port drivers or custom protocols needed.

Example — Straight key clean-up

Connect a straight key via any input method (keyboard, MIDI input, or audio). The app decodes the operator’s hand-timed morse into characters. MIDI output then replays those characters as perfectly timed dits and dahs at the configured Encoder WPM speed — effectively “cleaning up” the timing before it reaches the keying hardware.

7.5  Sidetone (Speaker Audio)

The sidetone is the audible tone you hear through your speakers or headphones when Morse code is being sent or received. The Active on selector controls which events produce a sidetone:

  • Both (default) — Plays for TX and RX events, so you always hear what is happening.
  • TX only — Plays only when you are sending (keyer, encoder, typed text).
  • RX only — Plays only for decoded incoming signals (audio inputs, Firebase RTDB).

Settings

  • Audio Device — Which speakers or headphones to use.
  • Channel — Left or Right.
  • Frequency — Tone pitch in Hz. 500–800 Hz is typical. Choose a frequency that is comfortable for long operating sessions.
  • Amplitude — Volume level. Start at 0.3–0.5 and adjust to taste.
  • Active on — TX only, RX only, or Both (see above).

7.6  Vibration (Haptic Feedback)

The vibration output mirrors the sidetone: the device vibrates while the key is down and stops when it goes up. Like the sidetone, the Active on selector controls which events trigger vibration:

  • Both (default) — Vibrates for TX and RX events.
  • TX only — Vibrates only when you are sending.
  • RX only — Vibrates only for decoded incoming signals.

Android only. The Vibration API (navigator.vibrate()) is supported in most Android browsers (Chrome, Firefox, Edge). iOS Safari does not support it, and desktop browsers silently ignore it. Vibration is disabled by default and can be enabled in the Outputs tab of Settings.

Enhanced Haptic Timing

Android vibration motors have a physical spin-up delay of 20–50 ms, which makes short dits imperceptible at higher speeds. When the Enhanced haptic timing option is enabled (default: on), the app sends a brief pre-pulse pattern that starts the motor spinning before the main pulse begins, and uses a short delayed cancel on key-up. This dramatically improves tactile clarity at 10+ WPM. You can disable it if you notice any timing artefacts on your device.

7.7  Loop Detection

When an output is configured to activate on RX signals, and the resulting morse is picked up by an input (e.g. via a microphone, CW audio input, or even a network round-trip through Firebase RTDB), a feedback loop can form. The same characters are decoded, sent back out, re-decoded, and the cycle continues endlessly.

Morse Code Studio includes an automatic loop-detection mechanism. It continuously compares the most recent characters sent to outputs with those arriving from inputs. When six or more consecutive characters match within a sliding five-second window, the application concludes that a loop is occurring and temporarily suppresses all output routing for an eight-second cooldown period. If the loop triggers again immediately after the cooldown expires, the cooldown increases linearly (up to a maximum of 32 seconds).

While a loop is suppressed a yellow warning banner appears at the top of the page with a brief explanation. You can dismiss the banner manually by clicking the × button, which also clears the suppression immediately.

Avoiding Loops

  • Use TX only on outputs that feed back into an input you are monitoring (e.g. if your radio's sidetone goes into the CW audio input, set the optocoupler or serial output to TX only).
  • When using Firebase RTDB, ensure the channel is configured so that received characters are not re-published back to the same channel.
  • When re-keying received morse through a WinKeyer or serial port, make sure the resulting signal does not reach an active input.
